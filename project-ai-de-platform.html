Why I built this

Most portfolio projects stop at the code. They demonstrate a concept — Kafka consumer, dbt model, LLM call — but don't show what production engineering actually looks like: tested code, repeatable CI, mocked dependencies, clear separation of concerns.

I built this platform to close that gap. Every repo runs a test suite in GitHub Actions. Every external service (Snowflake, OpenAI, Anthropic, Slack, Kafka) is mocked at the right boundary so tests are fast, deterministic, and infra-free. The architecture patterns are the same ones I'd apply in a production environment.

The goal: when a hiring engineer opens any repo in this platform, it should read like work that was already in production — not a demo built to impress.

Testing strategy

The constraint I set myself: every test must pass in GitHub Actions with no external infrastructure. No Snowflake cluster. No Kafka broker. No OpenAI credits. No Spark session. That meant building deliberate mock boundaries for each service.

Dependency How it's mocked Why this boundary
Anthropic / Claude API unittest.mock.patch on anthropic.Anthropic client, returns structured JSON via MagicMock Tests prompt construction and response parsing without API cost or rate limits
OpenAI embeddings Patched at openai.OpenAI client level; returns fixed-length float arrays Validates batching logic, retry behavior, and cache hit/miss without network calls
Snowflake connector Cursor mock with fetchall() returning fixtures; connection patched via conftest.py Exercises query construction and result mapping without a live warehouse
Slack webhooks requests.post patched at module level, response mocked to 200 OK Validates Block Kit payload structure and error handling without sending real messages
PySpark UDFs Pure Python unit tests on underlying functions; Spark context never instantiated Spark session startup adds 30+ seconds and requires JVM — not suitable for fast CI
Kafka producer confluent_kafka.Producer patched; produce() and flush() verified via call assertions Tests serialization and schema compliance without a live broker
PII secret (HMAC) conftest.py sets PII_TOKENIZATION_SECRET before module import to avoid EnvironmentError at load time Module raises at import if env var is missing — secret must be in scope before test collection

CI strategy

Every repo ships with a .github/workflows/ci.yml that runs on push and pull request to main. The Python repos test across both 3.11 and 3.12 via a matrix build.

Each CI workflow runs in sequence:

  • Install dependencies — lightweight installs only; no Spark, no Kafka cluster spun up
  • Run pytest with coverage — fails if coverage drops below 80%
  • flake8 lint — enforces line length and style consistency
  • mypy type check (where applicable) — strict mode on core modules
  • SQLFluff lint (dbt repo) — Snowflake dialect, dbt templater
  • Schema validation (dbt repo) — YAML parse check on schema.yml
The fintech-data-platform repo uses continue-on-error: true on SQLFluff deliberately — linting is enforced as a signal, not a hard block, because dbt templating in CI without a real Snowflake profile causes false negatives on {{ ref() }} resolution. The schema validation and non-empty SQL checks are hard failures.

What this proves

Building this platform required making the same decisions I'd make as a senior engineer on a real team: where to draw mock boundaries, how to handle import-time side effects in test environments, when to fail CI hard vs. soft, and how to structure repos so they're navigable to someone new.

  • I know where abstraction belongs and where it adds noise
  • I can make a test suite fast without sacrificing coverage of real behavior
  • I treat CI as a first-class deliverable, not an afterthought
  • I apply consistent patterns across different problem domains (streaming, LLM, analytics)

The repos are open. The CI logs are public. Read the tests — that's where the real engineering decisions live.
